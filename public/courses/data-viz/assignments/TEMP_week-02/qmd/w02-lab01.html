Working with Paths

Overview

Data analysis usually involves loading files into R from your local computer. This does not always go smoothly. In this activity, we’re going to learn a bit about how to use paths correctly to avoid some common errors.

To load a file, you need provide a correct path to its location on your computer. The path may be relative or absolute. An absolute (or “full”) path includes the root (top-level) directory of your file system and points to the same location regardless of your R session’s the current working directory.

Examples of absolute paths:

  • Windows: "C:/Users/czx412/Documents/deeply/nested/path/that/only/I/have/data/my_file.csv"
  • Mac: "/Users/czx412/Documents/another/ridiculous/path/that nobody else/can/use/data/my_file.csv"

A relative path starts from some working directory_, avoiding the need to provide the full, absolute path. For this to work properly, the program needs to know your starting point (i.e., the working directory).

Example of a relative path:

  • Any system: "data/my_file.csv"


There are a few special paths that are useful:

  • ~ is the user’s home directory. On this particular computer, ~ is equivalent to /Users/czx412. This will change depending on which computer I’m using.
  • .. is the parent directory. So ../my_file.csv refers to a file called my_file.csv in the directory that is one level above the working directory.

The key to getting paths to work in R is understanding R’s working directory.

What is your working directory at this very moment? You can use the getwd() function to find out.

getwd()
[1] "/Users/czx412/Github/data-viz/assignments/TEMP_week-02/qmd"

Common Problems

There are several problems that people commonly run into when trying to specify the right path:

Absolute paths don’t work on different computers

A file that exists in a certain place on your computer almost certainly does not exist in that same place on someone else’s computer, or on another computer that you own. Since it is unworkable to force people to put files in the same places, it is a good idea to use relative paths. If you share a project with somebody and use relative paths, then these relative paths should work any computer, regardless of where the project root directory is located. This activity is an example.

Not sharing with somebody at this moment? Don’t forget future-you! Current-you should should have an interest in helping future-you to work efficiently rather than sabotaging their work by filling your R scripts with what will eventually become broken file paths.


R Markdown vs R session or project root

When you’re working in a project, your working directory is the project’s root directory.

However, the default working directory for an R Markdown document is the directory in which that .Rmd document is stored. Suppose that you saved your R Markdown file in a directory called rmd that lives within the root directory of your project, and your data files in another sub-folder called data. Use the file explorer to verify that this is how I organized this project. It is generally a good idea to organize your files in this way. But the working directory of the R Markdown file is now different than your project root directory.

In addition, if you use the command setwd() in your R session, then the working directory of your current R session will differ from the working directory of your R Markdown document AND your project root. In brief, lots of potential for total chaos.



The ‘here’ package

The here R package is very simple and has only one goal: finding files in your project. It will always locate the files relative to your project root. When the package is first loaded, you’ll get a message telling you the location of the folder that is your project root.

library("here")
here() starts at /Users/czx412/Github/data-viz/assignments/TEMP_week-02

The here() function locates a file relative to your project’s root folder. This is how you use it:

# Create path to a csv file in project root folder
here("file_i_want.csv")
[1] "/Users/czx412/Github/data-viz/assignments/TEMP_week-02/file_i_want.csv"
# Create path to a csv file located in a subfolder called "data" in your project root folder
here("data/file_i_want.csv")
[1] "/Users/czx412/Github/data-viz/assignments/TEMP_week-02/data/file_i_want.csv"

Note that your file paths will (and should) look different from mine.

Why would you want to use this? Two main reasons:

  1. The same command will “just work” whether it is run from your console, an R source file, or while knitting an R Markdown document.

  2. It is not affected by your fiddling around with setwd() (which I suggest always avoiding if possible). Instead, it will figure out the top level of your current project, and locate the file from there.



Activity

As an example, in my personal project, this is how I would use the here() function to locate and read the file school_earnings.csv, which is located in a subfolder of my project called “data”.

First, note your current working directory. Note (and also verify) that this will give a different result if you run it in console vs. this .Rmd file!

getwd() # Also run this in console
[1] "/Users/czx412/Github/data-viz/assignments/TEMP_week-02/qmd"


From your R console’s perspective, your root directory is the project folder. From your R markdown file’s perspective, the root directory is the location of the .qmd file. Uh-oh!

Modify the command below so that it executes successfully for you in your R console (not when knitting). Note that I added the error=TRUE option to the R chunk to allow the document to knit, despite the error, and show the R error in the report.

school_earnings <- read.csv("<your relative path here>")
Warning in file(file, "rt"): cannot open file '<your relative path here>': No
such file or directory

Now modify the command below so that it works correctly when knitting but not in the R console. If it works, you should see no error message, and a table of values should appear below the R chunk. But it no longer runs properly in the R console. (Hint: you’ll need to navigate “up” from your .Rmd file’s working directory).

school_earnings <- read.csv("<your relative path here>")
Warning in file(file, "rt"): cannot open file '<your relative path here>': No
such file or directory
Error in file(file, "rt"): cannot open the connection

You might be tempted to solve this problem by providing the absolute path. Future-you is very disappointed 😢.

The following works on my office computer but not on my home computer, and definitely does not work on any of your computers (you should see an error, which does not appear for me).

# This works for me alone!
school_earnings <- read.csv("/Users/czx412/Github/data-viz/assignments/TEMP_week-02/data/school_earnings.csv")

Modify the absolute path so that it works for you.

# This should work for you alone. If successful, you should see no error below this box.
school_earnings <- read.csv("<your absolute path here>")
Warning in file(file, "rt"): cannot open file '<your absolute path here>': No
such file or directory

The code below should “just work” in any scenario. Verify that this works both in the knitted document and in your R console.

# Works both in console and when knitting
school_earnings <- read.csv(here("data/school_earnings.csv"))

Best practices

Here are my recommendations for avoiding path-related pain in R.

  1. Always work in a project.
  2. Always use relative paths.
  3. Never use the setwd() command.
  4. If your project folder is complex and organized into subfolders, use the here R package to refer to files relative to the project root folder.
