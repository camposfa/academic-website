Week 2, Lab 2: Files and Paths

Published

August 30, 2023


Data analysis usually involves loading files into R from your local computer. This does not always go smoothly. In this activity, we’re going to learn how to use paths correctly to avoid some common errors.


The Working Directory

The key to everything related to paths in R is the concept of a working directory.

Important

The working directory is the folder on your computer that your current R session will use for loading files into R or saving files from R, unless you explicitly command it to look elsewhere.

Getting paths to work properly in R requires an understanding your current R session’s working directory.

What is your working directory at this very moment? You can use the getwd() function to find out. This is going to be different on everybody’s computer, but here is mine:

getwd()
[1] "/Users/czx412/Library/CloudStorage/OneDrive-UniversityofTexasatSanAntonio/Work/UTSA/Teaching/Data Visualization/Data Visualization - Fall 2023/website/assignments/week-02"
Note

A path in R is stored as a string of characters, which you can tell because it is enclosed in quotes (").


Kinds of Paths

To refer to any document or folder in R, you need provide a correct path to its location on your computer. The path may be relative or absolute. What’s the difference?

Absolute Paths

Absolute paths start from the root (top-level) directory of your filesystem and include every step (sub-folder) you must take through the filesystem to reach the target location. The working directory is irrelevant to a properly specified absolute path. In other words, an absolute path points to the same location on a particular computer regardless of your R session’s current working directory.

Examples of made-up absolute paths:

  • Windows  : "C:\Users\czx412\Documents\deeply\nested\path\that\only\I\can\use\data\my_file.csv"
  • Mac   / Linux  : "/Users/czx412/Documents/deeply/nested/path/that/only/I/can/use/data/my_file.csv"

Note that absolute paths look different depending on the operating system. In Windows, they start with a drive letter followed by a colon, and in Mac/Linux they start with a forward slash.

Also, for weird historical reasons, paths in Windows traditionally use \, while Unix/Mac use / to separate the folder names. This difference adds an extra complication: \ is a special character in R, which means that the Windows path as written above would not work in R. Fortunately, most apps in Windows (including R) have added support for paths that use /:

  • Absolute path in Windows    that would work in R: "C:/Users/czx412/Documents/deeply/nested/path/that/only/I/can/use/data/my_file.csv".

You can still write your paths using \ if you’re running R on Windows, but because the single backslash is a special character, you need to use two of them each time:

  • Absolute path in Windows    that would work in R, as written by the utterly deranged: "C:\\Users\\czx412\\Documents\\deeply\\nested\\path\\that\\only\\I\\can\\use\\data\\my_file.csv".
Caution

The independence of absolute paths from the working directory concept might make you think absolute paths are always the way to go in R. And you’d be wrong. Avoid using absolute paths in R!

Absolute paths are a common source of problems.

Aside from their annoying length, absolute paths don’t work on different computers and tend to break over time. A file that exists in a certain place on your computer right now almost certainly does not exist in that exact same place on someone else’s computer, or on another computer that you use now or in the future. But if you use relative paths (see below) in a consistent way, then they should work other computers, regardless of where the project root directory is located in each different computer’s file system.

Not sharing code with anybody or using multiple computers right at this very moment? Don’t forget future-you! Current-you should should have an interest in helping future-you to work efficiently rather than sabotaging their work by filling your R scripts with what will eventually become broken file paths.


Relative Paths

Relative paths are typically far shorter and simpler. A relative path is relative to some assumed starting point, avoiding the need to provide the full, absolute path. For this to work properly, the program needs to know your assumed starting point: our old friend, the working directory.

Here’s an example of a relative path:

  • Any system: "data/my_file.csv"

Translation: in the working directory, look in a subfolder called "data" for a file called "my_file.csv".

Hence, the relative path "data/my_file.csv" would be interpreted in any R session as simply appending this to the end of the R session’s working directory, which in my case results in:

[1] "/Users/czx412/Library/CloudStorage/OneDrive-UniversityofTexasatSanAntonio/Work/UTSA/Teaching/Data Visualization/Data Visualization - Fall 2023/website/assignments/week-02/data/my_file.csv"


Special Characters

There are a few special characters that are useful when using relative paths:

  • ~ is the user’s home directory. On this particular computer, ~ is equivalent to “/Users/czx412”. This will change depending on which computer I’m using.
  • .. is the parent directory. So the path "../my_file.csv" would refer to a file called "my_file.csv" in the directory that is one level above the working directory.


Best Practices

The best and easiest way to avoid most problems related to files, paths, and working directories when using R is to use relative paths and always work in an RStudio project.

As we discussed in class, an RStudio Project is basically a folder in which you will keep all of your input data, R scripts, reports, and outputs related to a particular work project together in one place. When you create a new project, RStudio will do two important things:

  1. Create a .Rproj file that keeps track of all your project-specific settings. If you close your project, take a break for a while, and then come back to it, RStudio will use this .Rproj file to reopen the files you were working on, load your project-specific command history, reapply any settings, etc.

  2. Your working directory will be automatically set to the project root folder (that is, the location of the .Rproj file). You do NOT have to use the command setwd() to set your working directory, and indeed, you never should.


Avoiding Pitfalls

R session vs. Quarto/RMarkdown

Another sneaky source of problems, even when you’re working in a project and using relative paths, can occur when you are reading or writing files from a Quarto document (all of this is also true for R Markdown, but for the sake of brevity, I will only refer to Quarto below).

Specifically, problems often occur when your .qmd file is located somewhere other than your project root folder. And the simple reason is that a Quarto document functions temporarily as its own, self-contained R session while it is rendering. When this happens, the working directory for the rendering document becomes whatever folder the .qmd file lives in, regardless of the working directory in your “main” R session.

If the .qmd file is in the project root folder, there’s no problem, because the working directories for both the R session and rendering document are the same.

But if the .qmd file is located somewhere else—say, in a subfolder of your project, or in some other location entirely—then you can get problems. In such cases, a command to load data using a relative path that runs successfully in your R script (or when executed in your R session from the .qmd file itself) will fail during rending, and vice versa.


The ‘here’ package

The here R package was created in part to resolve this problem. It’s very simple and has only one goal: locating files in a project. It will always locate the files relative to your project root folder. When the package is first loaded, you’ll get a message telling you the absolute path to your project root.

library("here")
here() starts at /Users/czx412/Library/CloudStorage/OneDrive-UniversityofTexasatSanAntonio/Work/UTSA/Teaching/Data Visualization/Data Visualization - Fall 2023/website

The here() function locates a file relative to your project’s root folder, and converts this behind the scenes into a correct absolute path for whatever computer you’re currently using, regardless of whether it’s used in an R session or while rendering a Quarto document. This is how you use it:

# Specify path to a csv file in project root folder
here("file_i_want.csv")
[1] "/Users/czx412/Library/CloudStorage/OneDrive-UniversityofTexasatSanAntonio/Work/UTSA/Teaching/Data Visualization/Data Visualization - Fall 2023/website/file_i_want.csv"
# Specify path to a csv file located in a subfolder called "data" in your project root folder
here("data/file_i_want.csv")
[1] "/Users/czx412/Library/CloudStorage/OneDrive-UniversityofTexasatSanAntonio/Work/UTSA/Teaching/Data Visualization/Data Visualization - Fall 2023/website/data/file_i_want.csv"
Note

The file path that you actually provide to the here() function is always a simple relative path. The absolute paths generated by the function will (and should) look different in your document from mine and everyone else’s.

Why would you want to use this? Two main reasons:

  1. The same command will “just work” whether it is run from your console, an R source file, or while rendering a Quarto document.

  2. It is not affected by your fiddling around with setwd() (which I suggest always avoiding if possible). Instead, it will figure out the top level of your current project, and locate the file from there.
