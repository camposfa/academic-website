Working with Paths

Overview

Data analysis usually involves loading files into R from your local computer. This does not always go smoothly. In this activity, we’re going to learn a bit about how to use paths correctly to avoid some common errors.

To load a file, you need provide a correct path to its location on your computer. The path may be relative or absolute.

Absolute Paths

An absolute (or “full”) path includes the root (top-level) directory of your filesystem and points to the same location regardless of your R session’s the current working directory. Absolute paths start from your hard drive and include every step (subfolder) you must take through the filesystem to reach the target location. Where you are currently working is irrelevant to the absolute path.

Examples of absolute paths:

  • Windows: "C:\Users\czx412\Documents\deeply\nested\path\that\only\I\have\data\my_file.csv"
  • Mac/Linux: "/Users/czx412/Documents/another/ridiculously/long/path/that/only/I/can/use/data/my_file.csv"

Note that absolute paths look different depending on the operating system. In Windows, they start with a drive letter followed by a colon, and in Mac/Linux they start with a forward slash.

Also, paths in Windows use \, while Unix/Mac uses /. This difference adds an extra complication: \ is a special character in R, which means that the Windows path as written above would not work in R. Fortunately, R in Windows can understand paths that use / to separate the folders:

  • Absolute path in Windows that works in R: "C:/Users/czx412/Documents/deeply/nested/path/that/only/I/have/data/my_file.csv".

You can still write your paths using \ if you’re running R on Windows, but because the single backslash is a special character, you need to use two of them each time:

  • Absolute path in Windows that works in R, as written by the utterly deranged: "C:\\Users\\czx412\\Documents\\deeply\\nested\\path\\that\\only\\I\\have\\data\\my_file.csv".

Now that we’ve gone through all that: NEVER USE ABSOLUTE PATHS IN R!!

Why?

Aside from their annoying length, absolute paths don’t work on different computers and tend to break with time. A file that exists in a certain place on your computer almost certainly does not exist in that same place on someone else’s computer, or on another computer that you own or work on. Since it is not reasonable to force people to put files in the same places, it is a good idea to use relative paths. If you share a project with somebody and use relative paths, then these relative paths should work any computer, regardless of where the project root directory is located. This activity is an example.

Not sharing code with anybody or using on multiple computers at this time? Don’t forget future-you! Current-you should should have an interest in helping future-you to work efficiently rather than sabotaging their work by filling your R scripts with what will eventually become broken file paths.


Relative Paths

Relative paths are typically far shorter and simpler. A relative path is relative to some assumed starting point, avoiding the need to provide the full, absolute path. For this to work properly, the program needs to know your starting point, which in R is called the working directory.

Here’s an example of a relative path:

  • Any system: "data/my_file.csv"

The key to getting relative paths to work in R is understanding R’s working directory.

What is your working directory at this very moment? You can use the getwd() function to find out.

getwd()
[1] "/Users/czx412/Documents/Work/UTSA/Teaching/Data Visualization/Data Visualization - Fall 2023/website/assignments/TEMP_week-02"

There are a few special characters that are useful when using relative paths:

  • ~ is the user’s home directory. On this particular computer, ~ is equivalent to /Users/czx412. This will change depending on which computer I’m using.
  • .. is the parent directory. So ../my_file.csv refers to a file called my_file.csv in the directory that is one level above the working directory.


R Markdown vs R session or project root

When you’re working in a project, your working directory is the project’s root directory.

However, the default working directory for an R Markdown document is the directory in which that .Rmd document is stored. Suppose that you saved your R Markdown file in a directory called rmd that lives within the root directory of your project, and your data files in another sub-folder called data. Use the file explorer to verify that this is how I organized this project. It is generally a good idea to organize your files in this way. But the working directory of the R Markdown file is now different than your project root directory.

In addition, if you use the command setwd() in your R session, then the working directory of your current R session will differ from the working directory of your R Markdown document AND your project root. In brief, lots of potential for total chaos.
